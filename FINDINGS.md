# autoexperiment — findings

Sample run of the autoexperiment template against a remote `llama.cpp` +
`LiteLLM` endpoint serving two Gemma reasoning models on Apple Silicon.

- **Metric:** `tokens_per_second` = `completion_tokens / wall_time` (higher is better)
- **Branch:** `autoexperiment/apr21-1100`
- **Runs:** 60 (6 baselines + 54 explorations)
- **Artifacts:** `progress.log`, `results.tsv`, `results.html` (generated by `viz.py`)

## Result

| Phase                | `tokens_per_second` | Config                                                                          |
|----------------------|--------------------:|---------------------------------------------------------------------------------|
| Best baseline (qa)   |               85.56 | `supergemma4-26b` · qa · `max_tokens=200` · streaming                           |
| **Best overall**     |           **90.02** | `supergemma4-26b` · qa · `max_tokens=1024` · `top_k=1` · `temp=0` · non-stream  |

**Lift: +5.21%** over the best-matching baseline, **+18.47%** over the lowest baseline
(gemma4-26b summarization @ 76.00).

## What moved the number (kept changes)

1. **Model choice — `supergemma4-26b` over `gemma4-26b`.** Roughly +6 tok/s at
   matched settings. Biggest single lever in the search.
2. **`MAX_TOKENS=1024`.** Longer generations amortize prefill and let decode
   settle into its steady-state rate. Below 512 the prefill share dominates;
   above 1024 decode slows as the KV cache grows.
3. **Greedy fast path — `top_k=1` + `temperature=0`.** Small but reproducible
   gain; `llama.cpp` has a dedicated argmax path for k=1.
4. **Non-streaming requests.** Dropping SSE framing shaved a token or two
   per second off end-to-end time with no code complexity cost.

## What didn't work (notable discards)

- **`MAX_TOKENS` above 1024** (1100, 1200, 1500, 2048) — decode throughput
  drops as KV cache grows. Metal shows clear diminishing returns.
- **`MAX_TOKENS` between 400–900 and 1040–960** — all within noise of, or
  worse than, 1024. Extensively swept, no winner.
- **Sampler knobs beyond greedy** (`top_p=1.0`, `min_p=0.0`, different `seed`,
  `temp=1.0` with `top_k=1`) — no measurable effect.
- **`n=2` (parallel choices)** — dropped to ~67 tok/s. Server serializes the
  decodes and the usage accounting becomes unreliable.
- **Prompt changes** (summarization, open-ended, alt qa prompts) — qa remained
  fastest; summarization is EOS-early and has minimal prefix cache to exploit.
- **`cache_prompt=true`** — no effect (already the default).
- **System prompts, longer warmups, inter-trial sleeps** — no effect.
- **`NUM_TRIALS` changes** (3 / 6 / 7 / 8) — shifted the reported mean around
  within the noise floor but never moved the optimum.

## Measurement notes

- Trial-to-trial noise on this endpoint is roughly **±1.6 tok/s** on the best
  config. Improvements smaller than that should be treated as noise —
  several `keep` → `discard` flips in the log are explained by this alone.
- The LiteLLM proxy frequently buffers small SSE completions. When it does,
  `decode_tok_s` reports `0.0` and only the end-to-end figure is trustworthy.
  The experiment code handles this by always privileging end-to-end.
- `supergemma4-26b` emits more reasoning tokens than `gemma4-26b`, which
  helps the tokens/sec metric because decode is the fast phase — reasoning
  tokens stretch the steady-state decode window.

## Reproducing

```bash
export LITELLM_BASE_URL=…
export LITELLM_MASTER_KEY=…

uv sync
uv run harness.py       # connectivity + model check
uv run experiment.py    # single run with the winning config (HEAD)
uv run viz.py --open    # regenerate results.html and open it
```

The winning configuration is checked in at `HEAD` of the
`autoexperiment/apr21-1100` branch (`cf86f16`).
